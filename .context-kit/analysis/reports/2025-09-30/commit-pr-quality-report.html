Recommendations

High Priority

  1. Implement commit message hooks

    Use commitlint with husky to enforce conventional commits format:

    npm install --save-dev @commitlint/cli @commitlint/config-conventional husky
npx husky install
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit $1'
  2. Break large commits into smaller pieces

    Target: Keep commits under 200 lines when possible, max 500 lines for features

    • Split by logical functionality
    • Separate refactoring from feature additions
    • Keep test additions with the code they test

Medium Priority

  1. Consider feature branch workflow

    Even for solo projects, PRs provide:

    • Opportunity to review changes before merging
    • CI/CD validation gates
    • Better change documentation via PR descriptions
    • Practice for team collaboration
  2. Add breaking change indicators

    Use BREAKING CHANGE: footer in commit bodies for API changes:

    feat(api): redesign authentication flow

BREAKING CHANGE: Auth tokens now require refresh flow.
Old tokens will be invalidated after migration.

Low Priority

  1. Standardize scope naming

    Create a documented list of valid scopes to maintain consistency:

    • ui - UI components and styling
    • routing - Navigation and URL handling
    • context-kit - Context kit infrastructure
    • storybook - Storybook documentation
    • portfolio - Portfolio content
  2. Use conventional commit tools

    Consider CLI tools like commitizen for interactive commit message creation:

    npm install -g commitizen
commitizen init cz-conventional-changelog --save-dev --save-exact
